## Project Overview

Unikraft makes it easy to run existing applications in two ways:
1. **Musl libc support**

2. [**Binary-compatibility mode**](https://unikraft.org/docs/concepts/compatibility):
Unikraft can run unmodified Linux ELF binaries by using an ELF loader app and a system-call shim layer.
When a binary-compatible unikernel starts, the app-elfloader parses and maps the ELF segments into memory, then transfers control to the application entry point.
Whenever the application invokes a Linux system call, the syscall shim intercepts the call number and routes it to the corresponding Unikraft handler.
Unimplemented calls return ENOSYS, which many applications can handle or fake because Unikraft follows Linux’s x86_64 ABI closely.
This helps when running on hypervisors like KVM, QEMU, or Firecracker.

## Current Progress

After successful investigation of around 14 applications in the first 6 weeks, I started contributing to the system call layer of Unikraft, from week 7.
In those first 6 weeks, I found three features blocking support for many applications:
1. **No support for writable MAP_SHARED**:
    - `deno`, `netdata`, `prometheus`, `rabbitMQ`, `tomcat`

2. **No support for NETLINK sockets**
    - `consul`, `etcd`, `influxDB`, `minio`, `netdata`, `rabbitMQ`, `syncthing`, `tomcat`, `vault`, `wireguard`

3. **No support for clone without CLONE_VM**
    - `gitea`

In the last three weeks I have been working on implementing a workaround for writable `MAP_SHARED` (see draft PR: https://github.com/unikraft/unikraft/pull/1679).
The workaround converts writable shared mappings into private ones.
This is useful only when neither persistence, nor synchronization with read / write is required for the mapped file.
From the applications which I tested earlier, this workaround should work for `deno`, `prometheus` and `netdata`.
It won't help workloads that rely on writeback / synchronization, such as `rabbitMQ` and `tomcat`.
I tested the workaround on `prometheus` and it worked as expected.
